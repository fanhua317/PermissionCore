# PermaCore IAM 性能测试报告

> 测试时间：2026-07-10 至 2026-07-11
>
> 结论适用范围：本机、固定容器资源和确定性合成数据；这是容量分类结果，不是生产 SLA。
>
> 数据口径：只引用 `performance/results/` 中可复现的持续采样结果，不使用瞬时峰值。

## 1. 结论摘要

- 在 10k 数据集上，读混合场景的最高健康持续档由基线 100 RPS 提升到优化后 200 RPS；200 RPS 三轮中位实际吞吐 199.83 RPS，P95 16.52 ms，P99 28.38 ms，中位错误率 0。
- 在 100k 数据集上，优化前 50 RPS 档只能完成 23.43 RPS，P95/P99 为 8.90/11.59 s；优化后同档实际吞吐 49.91 RPS（2.13 倍），P95/P99 降至 232.45/432.52 ms。但 P95 仍高于 200 ms 健康线，且三轮波动明显，因此 **100k 读混合没有健康档，不把 50 RPS 写成可承诺容量**。
- 100k 并发角色分配由单次基线 31.85 ops/s、P95 964.53 ms 提升到三轮中位 297.22 ops/s、P95 141.70 ms、P99 228.19 ms；中位错误率 0.313%。由于基线写场景只有一次，该提升只能表述为“单次基线对三轮复测”，不能伪装成两侧均三轮的严格 A/B。
- 100k 全局权限变更从单次基线 0.390 mutation/s、P95 3.11 s，改善到三轮中位 7.305 mutation/s、P95 102.60 ms、P99 116.44 ms；根因是把每次扫描并更新 100k 用户改为单例 `global_auth_version + 1`。
- 30 分钟稳定性测试以 10k 健康档 200 RPS 的 70%（140 RPS）运行：实际 139.95 RPS，P95/P99 13.93/19.66 ms，错误率 0.00159%，没有连接池排队或持续资源恶化。随后 2 倍档 spike 全程错误率 0，并在回落阶段恢复到低延迟、零 Hikari pending。

## 3. 方法与判定口径

- 读阶梯每档先预热 30 秒，再有效采样 120 秒，独立重复 3 次。10k 测试 50/100/200/400/800 RPS；100k 基线测试 50/100，优化后补测 10/20/50/100 RPS。
- Login 使用 1/5/10/20/40 个恒定 VU，各运行 120 秒并重复 3 次；每 iteration 是一次 BCrypt 登录，TPS 等于请求率。
- Auth 使用相同 VU 与时长；每个完整循环包含 `info`、会话角色、Refresh 原子轮换和旧 Refresh 重放 401，共约 4 个 HTTP 请求。报告以完整循环 TPS 为主，不把 HTTP RPS 冒充业务 TPS。
- 写竞争使用 20 VU、120 秒，对不同目标用户分配角色；全局权限使用 1 VU 顺序执行 10 次权限变更，每次因旧 Token 被撤销而重新登录。
- 三次统计单元统一写成 `中位数 [最小值–最大值] / CV`；中位数取排序后 `floor(n/2)`，CV 为总体标准差除以均值。单次基线不计算范围和 CV。
- 健康档必须同时满足：错误率 `<1%`、P95 `≤200 ms`、P99 `≤500 ms`、Actuator `process_cpu_usage` 没有连续 3 个采样点 `>85%`、没有连续 3 个 Hikari pending。dropped iteration 也作为未达到目标吞吐的证据。

### 4.3 100k `%term%` 模糊搜索限制

100k 优化复测的 MySQL digest 显示，`username LIKE '%term%' OR nickname LIKE '%term%'` 的 COUNT 执行 151 次、检查 15,100,000 行，平均 113.23 ms；即每次仍扫描约 100k 行。对应分页取 ID 查询平均 112.41 ms，虽然只检查 9,060 行并返回 3,020 行，但前导通配符无法由普通 B-Tree 消除。

因此 100k 读混合的剩余瓶颈是被有意保留的 contains-match 语义，而不是 Redis、JVM 堆或连接池配置。若产品允许改变搜索语义，下一阶段应选前缀匹配；若必须保留任意包含匹配，应采用 MySQL FULLTEXT/n-gram 或专用检索服务并单独重建容量基线，不能通过盲目扩大 Hikari 掩盖问题。

### 6.2 SQL 与锁证据

- 基线写竞争的 `SELECT id FROM sys_role ORDER BY id FOR UPDATE` 执行 3,876 次，累计 2,239.96 s，平均 577.91 ms，证明全局角色图排他锁把不同用户写入串行化。
- 优化后锁顺序固定为“角色图共享锁 → 目标用户行锁”；三轮中位写入接近 300 ops/s。代表轮仍有 8,488 次 InnoDB row-lock wait、累计 211.55 s，但 Hikari pending 最大为 0，P99 仍为 228.19 ms，说明竞争被限制在相同目标用户/行，而不是全图。
- 基线全局权限变更的 `UPDATE sys_user SET auth_version=auth_version+1` 执行 10 次、检查 1,000,000 行，平均 1,831.92 ms。优化后该 SQL 消失，改为单例全局版本 O(1) 递增；代表轮 COMMIT 平均 12.43 ms，授权状态联表读取平均 0.447 ms。
- 用户状态、密码、用户角色和登出仍递增个人版本；角色、权限、继承和 SoD 变更递增全局版本。没有使用授权缓存、Pub/Sub 或 Redis 故障降级来换 TPS。

| 优化 10k / 健康 200 RPS / r2 | 1,311.8 QPS；Threads_running 3/4；slow/lock/disk-temp 增量均 0；buffer hit 100% | pending 0；active 最大 2；连接获取均值 0.001 ms | 进程 CPU 17.4%/19.3%；堆 413/599 MiB；89 次 GC 共 451 ms | ops 199/200；内存约 1.14 MiB；blocked/rejected 均 0 |
| 优化 10k / 30m soak | 951.8 QPS；Threads_running 2/4；slow/lock/disk-temp 增量均 0；buffer hit 100% | pending 0；active 最大 3；连接获取均值 0.008 ms | 堆中位/最大 351/656 MiB；199 次 GC 共 601 ms；线程最大 93 | ops 140/143；内存固定约 1.106 MiB；blocked/rejected 均 0 |

基线 100k 的 buffer hit 已是 100%，却出现 MySQL 两核饱和、Hikari 排队和大量慢 SQL，说明核心问题是扫描行数和查询形状，不是 buffer pool 太小。优化后没有扩大 Hikari（仍为 20）；因为 pending 已归零且 MySQL CPU 才下降，满足“不用连接池掩盖慢 SQL”的约束。

MySQL digest 的直接变化包括：基线深分页全列查询 178 次检查 35,600,000 行、平均 3,386.67 ms；优化后先在覆盖索引中取 ID、再回表读取展示列。日志增加 `(time DESC, id DESC)` 稳定索引，recent 查询不再触发无意义 COUNT；Dashboard 六项统计合并为一次往返；操作日志列表不再读取大 JSON/参数字段。

## 8. 优化清单与保留门槛

| 改动 | 证据与决定 |
|---|---|
| 用户普通/深分页改为 count + 覆盖索引取 ID + 回表，列表去掉密码等无用列 | 10k 健康档提高 2 倍；100k 同档延迟大幅下降，保留 |
| 用户角色反向索引、角色继承反向索引 | 支撑反向查询和递归入口；SQL digest/回归无退化，保留 |
| 登录/操作日志稳定时间索引、recent 专用查询、操作日志瘦列 | 基线日志查询曾检查数千万行；优化后不再进入最高总耗时项，保留 |
| Dashboard 聚合单次查询 | 减少六次应用往返；代表 digest 平均 13.23 ms，保留 |
| 角色图共享锁 + 目标用户行锁 | 写 TPS 约 9.33 倍且 P99 降低，保留 |
| 单例 `global_auth_version` | 全局 mutation TPS 约 18.74 倍且保持提交后立即撤销，保留 |
| admin JWT 通配权限紧凑化和普通 Token 字节预算 | 解决请求头可用性并保持动态权限；属于正确性/安全性改动，保留但不算吞吐收益 |
| 扩大 Hikari、降低 BCrypt、权限短时缓存、Pub/Sub 最终一致性 | 均未采用 |

性能专项改动的保留门槛是三轮中位至少改善 10%，且 P99、错误率和授权语义不明显退化。Login/Auth 的混合结果未达到统一 10% 门槛，所以不声称性能收益；安全与正确性修复不受该性能门槛支配，但必须通过功能回归。

压测后还收口了 Dashboard `del_flag`、日志按 `id` 稳定排序、批量删除排序锁和角色新增图锁四项性能中性正确性修正。它们不改变本报告所测核心 SQL/写场景语义，也不计入已有性能增益。最终审阅镜像已通过 65 项后端测试，并以相同资源、10k 数据、200 RPS、30 秒预热、120 秒采样重跑三次：实际 RPS 中位 199.790，P95 中位 15.890 ms，P99 中位 23.447 ms，错误率三轮均为 0，确认关键健康档仍成立（`20260711-032837-10k`）。

## 9. 30 分钟稳定性与 spike 恢复

### 9.1 30 分钟 soak

- 负载：10k 数据，140 RPS（200 RPS 健康档的 70%），持续 30 分钟。
- 结果：251,969 次 iteration，实际 139.945 RPS；dropped 32；P50/P95/P99 为 7.44/13.93/19.66 ms；错误率 0.001587%（约 4 个请求），检查通过率 99.9984%。
- 首/中/末 5 分钟堆内存中位为 340.6/365.8/356.0 MiB，末段没有单调增长；线程中位 91/91/91；Redis used_memory 始终约 1.106 MiB；Hikari pending 全程为 0。
- 中/末 5 分钟进程 CPU 中位 11.0%/10.8%，MySQL Threads_running 中位均为 2。首段单点高 CPU 属于启动与预热，不作为持续容量数字。

### 9.2 2 倍档 spike 与恢复

脚本先以 140 RPS 运行 30 秒，再在 60 秒内升到 400 RPS，随后用 2 分钟回落到 140 RPS。全程实际平均 250.82 RPS，错误率 0，P95/P99 为 18.95/95.65 ms，Hikari pending 始终为 0。

恢复后半段约 60 秒的服务端采样为：平均约 194.5 RPS、平均服务端耗时 5.74 ms、进程 CPU 中位 18.5%、Docker 后端 CPU 中位 59.6%、MySQL Threads_running 最大 4。最后一个完整负载样本约 157.9 RPS 时，平均服务端耗时 5.48 ms、进程 CPU 12.4%；最终 graceful-stop 的不完整样本不计入恢复能力。未观察到连接池积压或错误率尾随。

## 10. 双节点一致性

最终审阅镜像有效结果：`20260711-034209-10k/consistency.json`。

| 验证 | HTTP |
|---|---:|
| 节点 A 签发 Token，在节点 B 读取 | 200 |
| 节点 B 修改权限 | 200 |
| 修改后旧 Token 在节点 A 使用 | 401 |
| 节点 B 异地登录后，节点 A 的旧会话 | 401 |
| 节点 B 新 Token 在节点 A 使用 | 200 |

结论：共享 Redis sessionId 与数据库个人/全局授权版本在两节点间立即生效；没有依赖本地缓存或 Pub/Sub 广播。

## 11. 工具侧事故与判废

以下结果不进入性能统计，且不能伪装成后端性能失败：

- `20260710-215958-100k`：完整基线 Login 已在 `212616` 产生有效结果；随后启动 Auth 时 Docker CLI 的 Go runtime 因主机提交内存不足失败，只留下空目录。现场 `lghub_agent` 的 PagedMemorySize 约 6.69 GB，临时结束 Logitech G HUB agent/tray 后不再复现，测试结束恢复该工具。后续 runner 对短命 Go/Docker 进程设置 `GOMAXPROCS=2`、`GOMEMLIMIT=128MiB`、`COMPOSE_PARALLEL_LIMIT=2`。
- `20260710-230653-100k`：Auth 1/5 VU 共 6 个摘要完整，予以保留；之后 Docker CLI 再次提交内存失败，未产生的档位不参与统计。

## 13. 简历安全口径

推荐写法：

> 在 4 vCPU/2 GiB 后端、2 vCPU/2 GiB MySQL 的隔离 Docker 环境中构造 10 万用户、约 30 万用户角色关系及 40 万审计日志的确定性数据集，使用 k6 三轮复测优化分页/日志 SQL、授权锁粒度和全局 Token 撤销；10k 数据下读混合场景稳定支撑 200 RPS（P95 16.5 ms、P99 28.4 ms、错误率中位 0），并完成 30 分钟 140 RPS 稳定性测试（P95 13.9 ms、P99 19.7 ms、错误率 0.0016%）；100k 数据下并发角色分配三轮中位 297 ops/s（P95/P99 141.7/228.2 ms、错误率中位 0.31%），全局权限变更三轮中位 7.30 ops/s（P95/P99 102.6/116.4 ms、错误率 0）。

不得写成：

- “100k 用户稳定 50/100 RPS”——本轮 100k 读混合没有通过 P95 健康线。
- “单机 2500+ TPS、2.45 ms、提升 7 倍”——当前镜像和固定资源下没有相应复现证据。
- “写吞吐稳定提升 9.33 倍”而不说明基线仅一次——严格材料中应附该限制。
- “生产 SLA”或“生产实测”——本报告是本地隔离、合成数据、固定资源下的容量分类。
