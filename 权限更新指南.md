# PermaCore 权限初始化与更新指南

本指南用于空数据库初始化、已有数据库补齐内置权限，以及排查管理员权限异常。数据库结构和权限数据只有一套事实源，禁止复制旧版手写 INSERT。

## 1. 唯一受支持的事实源

空数据库顺序固定为：

1. src/main/resources/db/schema.sql
2. src/main/resources/db/init-permissions.sql

职责：

- schema.sql：创建 permacore_iam 和全部表结构。
- migrations/20260710_add_auth_version.sql：把旧数据库升级到当前 auth_version 契约，并幂等补齐角色关联外键与“禁止自继承”检查；若存在孤儿关系会明确失败，需先修复数据。它不是空库初始化的替代品。
- init-permissions.sql：补齐内置部门、角色、权限、角色继承和 SoD 约束；不创建 admin，也不绑定用户角色。
- migrate-database.ps1：已有数据库执行上述 20260710 加固 migration 的 PowerShell 包装器。
- update-permissions.ps1：已有数据库执行 init-permissions.sql 的 PowerShell 包装器，不是另一份初始化逻辑。

不受支持的方式：

- 文档中复制出来的零散 INSERT。
- update-permissions.bat 中的固定密码流程。
- /api/admin/db/* 一类内部修复接口。
- 启动时无版本地重写角色、权限名称或状态。
- 用 Redis FLUSHDB/FLUSHALL 代替数据库更新和应用缓存失效。

## 2. 执行前检查

1. 确认当前目录是 PermissionCore 项目根目录。
2. 确认目标数据库主机和 permacore_iam 实例。
3. 备份数据库。
4. 记录更新前 Git 提交。
5. 检查是否存在自定义 API 权限（resource_type=3）。
6. 用第 8 节的检测 SQL 记录 ROLE_USER、ROLE_GUEST 的现有管理授权和 ROLE_AUDITOR 的非基线授权；脚本只自动撤销旧版本写入普通用户/访客的六项已知过宽基线，其他关联需人工审阅。
7. 停止权限管理写操作，避免更新期间并发修改。

备份示例：

    mysqldump --host=localhost --port=3306 --user=root --password --single-transaction --result-file=.\permacore_iam-before-permission-update.sql permacore_iam

`--result-file` 让 mysqldump 直接写入字节，避免 Windows PowerShell 5.1 的 `>` 把输出转成 UTF-16。备份文件可能包含用户和权限数据，不得提交到 Git。

## 3. 空数据库初始化

在项目根目录启动 MySQL 客户端：

    mysql --host=localhost --port=3306 --user=root --password

依次执行：

    SOURCE src/main/resources/db/schema.sql;
    SOURCE src/main/resources/db/init-permissions.sql;

不能只执行 init-permissions.sql；空数据库还没有 sys_user、sys_role、sys_permission 等表。当前 schema.sql 已直接包含 auth_version，空库不需要再跑 20260710 migration。

项目不提供默认数据库密码、JWT 密钥或管理员密码。表和权限基线完成后，首次管理员由应用 bootstrap 创建：

    APP_BOOTSTRAP_ENABLED=true
    APP_BOOTSTRAP_ADMIN_PASSWORD=<8-72 位的首次管理员密码>

创建成功后关闭 APP_BOOTSTRAP_ENABLED，并从环境中移除 APP_BOOTSTRAP_ADMIN_PASSWORD。

## 4. 已有数据库升级

2026-07-10 之前创建的数据库缺少 sys_user.auth_version。必须在启动新版应用前严格按“备份 → migration → init 权限 → 启动新版”执行。Windows PowerShell：

    .\migrate-database.ps1
    .\update-permissions.ps1

两个入口相互独立，都会先显示高风险目标确认并交互式读取 MySQL 密码。migrate-database.ps1 只执行幂等结构迁移；它成功后才能运行 update-permissions.ps1。两者都以原始 UTF-8 字节写入 mysql 标准输入，可安全处理 Windows PowerShell 5.1 和带 BOM 的 SQL。若脚本找不到 mysql：

1. 把 MySQL bin 目录加入当前 PATH。
2. 重新打开终端。
3. 运行 mysql --version 确认。

也可以通过 `-MySqlCommand 'C:\path\to\mysql.exe'` 显式指定客户端路径；不要把密码放进该参数或命令行。

不要为了省事把密码写回 bat、ps1、application.yml 或命令行。

## 5. Docker 数据卷

MySQL 镜像只在空数据卷首次创建时自动执行初始化目录中的 SQL。已有卷不会因为 docker compose up --build 自动重放 schema、migration 或权限脚本。

先备份并停止 frontend/backend 写入，再在项目根目录依次执行 migration 和 init 权限。PowerShell 或 Bash 均先原样复制文件，再在容器内重定向，避免文本管道改变 SQL 字节：

    docker compose cp .\src\main\resources\db\migrations\20260710_add_auth_version.sql mysql:/tmp/20260710_add_auth_version.sql
    docker compose exec -T mysql sh -c 'MYSQL_PWD="$MYSQL_ROOT_PASSWORD" exec mysql -uroot --default-character-set=utf8mb4 --binary-mode=1 permacore_iam < /tmp/20260710_add_auth_version.sql'
    docker compose exec -T mysql rm -f /tmp/20260710_add_auth_version.sql

    docker compose cp .\src\main\resources\db\init-permissions.sql mysql:/tmp/init-permissions.sql
    docker compose exec -T mysql sh -c 'MYSQL_PWD="$MYSQL_ROOT_PASSWORD" exec mysql -uroot --default-character-set=utf8mb4 --binary-mode=1 permacore_iam < /tmp/init-permissions.sql'
    docker compose exec -T mysql rm -f /tmp/init-permissions.sql

若 MySQL 已安全绑定到 127.0.0.1，也可以依次运行 .\migrate-database.ps1 和 .\update-permissions.ps1。

不要通过 docker compose down -v 解决普通权限更新问题；该命令会删除整个数据库卷。

## 6. 更新脚本必须保持的边界

审阅或修改 init-permissions.sql 时必须满足：

- 可重复执行，不因唯一键重复而失败。
- 只修正明确列出的内置 perm_key。
- 不得把全部非菜单权限统一改成 resource_type=2。
- 不得改变用户自建 API 权限的 resource_type=3、父节点、名称或状态。
- ROLE_USER、ROLE_GUEST 默认不授予后台管理查询权限；ROLE_AUDITOR 只使用明确内置只读白名单，不能用 LIKE '%:query' 自动吸收未来业务权限。
- 不得重新启用管理员主动停用的自定义角色或权限。
- 失败时事务回滚，不返回“部分成功”。

如果更新后自定义 API 权限变为按钮类型，或普通用户突然获得新业务查询权限，应立即停止发布、恢复备份并检查 SQL 的全表 UPDATE/LIKE 授权。

## 7. Token 撤销与重新登录

`init-permissions.sql` 会在同一事务内全局递增 `sys_user.auth_version`。事务成功提交后，旧 access/refresh token 会立即被数据库门禁拒绝，不依赖本地缓存广播，也不需要靠重启才能完成撤销。正确顺序是：

1. 已有库先执行 20260710 migration，再执行 init-permissions.sql。
2. 部署并启动全部新版后端实例，不能只升级其中一台。
3. 让受影响用户重新登录并获取新 token。
4. 验证新的 JWT 权限集合，并确认旧 access/refresh token 不能继续使用。

Redis 中遗留的旧会话键不会绕过 `auth_version` 数据库门禁，无需清除。`FLUSHDB`/`FLUSHALL` 可能删除共享 Redis 数据，禁止作为权限更新步骤。任何绕过规范脚本、直接修改用户角色、角色继承、角色权限、SoD、角色或权限状态的 SQL，也必须在同一事务中递增受影响用户的 `auth_version`；无法可靠确定范围时应全局递增。

## 8. 验证 SQL

确认结构迁移已完成：

    SELECT COLUMN_NAME, COLUMN_TYPE, IS_NULLABLE, COLUMN_DEFAULT
    FROM information_schema.COLUMNS
    WHERE TABLE_SCHEMA = 'permacore_iam'
      AND TABLE_NAME = 'sys_user'
      AND COLUMN_NAME = 'auth_version';

结果必须恰好一行，类型为 bigint、不可为空且默认值为 0；否则不要启动新版后端。

查看 admin 角色：

    SELECT r.id, r.role_key, r.role_name, r.status
    FROM sys_user u
    JOIN sys_user_role ur ON ur.user_id = u.id
    JOIN sys_role r ON r.id = ur.role_id
    WHERE u.username = 'admin' AND u.del_flag = 0;

查看管理员权限：

    SELECT p.perm_key, p.perm_name, p.resource_type, p.status
    FROM sys_role r
    JOIN sys_role_permission rp ON rp.role_id = r.id
    JOIN sys_permission p ON p.id = rp.permission_id
    WHERE r.role_key = 'ROLE_ADMIN'
    ORDER BY p.parent_id, p.sort_order, p.id;

检查自定义 API 权限类型：

    SELECT id, perm_key, perm_name, resource_type
    FROM sys_permission
    WHERE resource_type = 3
    ORDER BY id;

检查 ROLE_USER、ROLE_GUEST 的全部管理授权，以及 ROLE_AUDITOR 超出当前内置基线的授权：

    SELECT r.role_key, p.id AS permission_id, p.perm_key, p.perm_name,
           p.resource_type, p.status
    FROM sys_role_permission rp
    JOIN sys_role r ON r.id = rp.role_id
    JOIN sys_permission p ON p.id = rp.permission_id
    WHERE r.del_flag = 0 AND p.status = 1 AND (
      r.role_key IN ('ROLE_USER', 'ROLE_GUEST')
      OR (r.role_key = 'ROLE_AUDITOR' AND p.perm_key NOT IN (
        'system:user:query', 'system:role:query', 'system:permission:query',
        'system:sod:query', 'system:dept:query', 'system:log', 'system:log:query'
      ))
    )
    ORDER BY r.role_key, p.perm_key;

执行 init-permissions.sql 前后都保存该查询结果并比较。脚本会撤销旧版本曾自动授予 ROLE_USER/ROLE_GUEST 的六项内置管理查询权限；其他结果不等于漏洞，需逐条向权限负责人确认来源和业务必要性，只有确认属于历史误授权后才通过受审计的管理流程撤销，初始化 SQL 不自动删除其他历史或自定义关联。

检查关键内置权限：

    SELECT perm_key, resource_type, parent_id, status
    FROM sys_permission
    WHERE perm_key IN (
      'admin:*',
      'system:user',
      'system:user:query',
      'user:add',
      'user:edit',
      'user:delete',
      'user:assignRole',
      'user:resetPassword',
      'system:sod',
      'system:sod:query',
      'sod:add',
      'sod:edit',
      'sod:delete'
    )
    ORDER BY perm_key;

## 9. 接口验证

重启并重新登录后至少验证：

- GET /api/auth/info 返回当前有效权限。
- 用户列表可访问。
- 具有权限的管理员可以新增、编辑、删除、分配角色和重置密码。
- 角色权限分配与继承配置正常。
- SoD 页面可查询，SSD/DSD 拒绝路径生效。
- 自定义 API 权限仍为 resource_type=3。
- 普通角色没有因本次更新获得无关业务权限。

测试脚本不得把完整 accessToken 打印到共享日志。

## 10. 常见失败

### Table does not exist

原因：空数据库只执行了 init-permissions.sql。

处理：先恢复或执行 schema.sql，再执行 init-permissions.sql。

### Duplicate entry

原因：使用了旧版手写 INSERT 或脚本失去幂等条件。

处理：回滚事务，恢复 canonical 脚本；不要继续忽略错误。

### 更新后仍然 403

依次检查：

1. SQL 是否完整提交。
2. 角色和权限是否启用。
3. admin 是否绑定 ROLE_ADMIN。
4. 目标权限是否绑定角色。
5. 所有后端实例是否重启。
6. 用户是否重新登录。
7. 当前会话是否激活了正确角色。
8. SSD/DSD 是否阻止当前组合。

### 管理员不存在

不要在 SQL 中写固定密码哈希或管理员角色绑定。使用一次性 bootstrap，并提供 APP_BOOTSTRAP_ADMIN_PASSWORD；密码必须为 8-72 位。bootstrap 会创建或恢复 admin 并补齐 ROLE_ADMIN 绑定，成功后应立即关闭。
