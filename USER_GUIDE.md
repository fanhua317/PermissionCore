# PermaCore IAM 用户操作手册

本文面向系统管理员和普通使用者，说明当前界面的实际操作。部署、数据库和 Docker 配置请分别查看 README.md、DOCKER_GUIDE.md 与 权限更新指南.md。

## 1. 访问与登录

| 运行模式 | 前端地址 |
|---|---|
| 本地开发 | http://localhost:5173 |
| Docker | http://localhost |

后端默认地址为 http://localhost:54321。

首次管理员用户名为 admin，密码由部署者在首次启动时设置。项目没有默认管理员密码；不要尝试旧文档中的固定口令。

登录步骤：

1. 打开对应前端地址。
2. 输入用户名和密码。
3. 点击“登录”。

为避免用户名枚举，用户名不存在和密码错误统一提示“用户名或密码错误”。账号被禁用时联系系统管理员。

密码必须为 8-72 位。管理员首次登录后应立即从右上角个人菜单修改首次密码。

## 2. 权限如何生效

系统使用 RBAC3：

- 用户先被分配角色。
- 角色获得直接权限，也可继承父角色权限。
- SSD 在分配角色时阻止静态互斥组合。
- DSD 在会话激活角色时阻止动态互斥组合。
- 页面、按钮和后端接口都由当前会话的有效权限控制。

看不到菜单或按钮通常不是界面故障，而是当前激活角色不具备对应权限。

### 当前激活角色

登录后，顶栏显示当前激活角色。点击后可以勾选本次会话需要使用的角色：

1. 系统只显示当前用户已分配且启用的角色。
2. 保存时检查 DSD。
3. 保存成功后刷新访问令牌、菜单和路由。
4. 如果新权限不能访问当前页面，系统返回控制台。

角色切换不会改变管理员给用户分配的角色，只改变当前会话激活的子集。

## 3. 仪表盘

仪表盘展示：

- 有效用户数。
- 角色数。
- 权限点数。
- 今日登录次数。
- 最近操作记录。

快捷卡片只负责跳转。用户是否可以执行目标页面中的操作，仍由权限决定。

## 4. 用户管理

路径：侧边栏 → 系统管理 → 用户管理。

### 查看与搜索

- 按用户名或昵称搜索。
- 使用分页器切换页码和每页数量。
- 操作按钮会根据当前权限显示或隐藏。

### 新增用户

点击“添加用户”，填写：

| 字段 | 必填 | 说明 |
|---|---|---|
| 用户名 | 是 | 全局唯一，建议 3 至 20 个字符 |
| 昵称 | 是 | 页面显示名称 |
| 密码 | 是 | 8-72 位 |
| 邮箱 | 否 | 必须符合邮箱格式 |
| 手机号 | 否 | 联系电话 |
| 部门 | 否 | 所属部门 |
| 状态 | 是 | 正常或禁用 |

不要为多个用户设置相同初始密码。创建后通过安全渠道把密码交给本人，并要求尽快修改。

### 编辑、启停与删除

- “编辑”用于修改昵称、邮箱、手机号、部门和状态。
- 状态开关可禁用或启用账号。
- “删除”执行系统定义的删除流程。
- “批量删除”只适用于用户管理页中已勾选的用户，不代表其他模块都支持批量删除。

删除前先确认用户是否仍关联角色、日志或业务记录。不要把删除当作临时禁用。

### 分配角色

1. 点击“分配角色”。
2. 勾选角色。
3. 保存。

保存时系统校验 SSD；出现冲突时应调整角色组合或 SoD 策略，不要绕过校验直接改数据库。

### 重置密码

具有 user:resetPassword 权限的管理员可点击“重置密码”。新密码必须为 8-72 位，重置后应通知用户立即修改。

## 5. 角色管理

路径：侧边栏 → 系统管理 → 角色管理。

### 角色字段

| 字段 | 必填 | 说明 |
|---|---|---|
| 角色编码 | 是 | 唯一标识，建议使用 ROLE_ 前缀，如 ROLE_FINANCE |
| 角色名称 | 是 | 面向使用者的名称 |
| 描述 | 否 | 角色职责与适用范围 |
| 状态 | 是 | 启用或停用 |

不要使用 admin、user 等不带 ROLE_ 前缀的旧示例作为新角色编码。

### 分配权限

1. 点击“分配权限”。
2. 在权限树中勾选权限。
3. 保存。

分配前遵循最小权限原则。菜单权限决定页面入口，按钮权限决定写操作，API 权限用于后端资源控制。

### 配置继承

1. 点击“继承配置”。
2. 选择当前角色的父角色，可多选。
3. 保存。

当前角色是子角色，会获得所选父角色的有效权限。系统拒绝自继承和形成环的继承关系。

示例：

    ROLE_USER
      └─ ROLE_MANAGER

表示 ROLE_MANAGER 继承 ROLE_USER，不是双向继承。

## 6. 权限管理

路径：侧边栏 → 系统管理 → 权限管理。

### 权限类型

| 类型 | resource_type | 用途 |
|---|---:|---|
| 菜单 | 1 | 页面入口 |
| 按钮 | 2 | 页面操作 |
| API | 3 | 后端资源 |

### 命名约定

当前内置权限使用：

| 用途 | 示例 |
|---|---|
| 菜单 | system:user |
| 查询 | system:user:query |
| 写操作 | user:add、user:edit、user:delete |
| 超级权限 | admin:* |

不要混用旧演示接口中的 user:view、user:create 与当前内置命名。权限标识一旦投入使用应保持稳定，显示名称可以单独调整。

### 新增与编辑

1. 点击“添加权限”或目标行的“编辑”。
2. 填写权限标识、名称、类型、父级、排序和状态。
3. 保存。

API 权限应保持 resource_type=3。执行权限更新后，如自定义 API 权限被改成按钮类型，应停止发布并按 权限更新指南.md 排查。

删除父权限前先检查子权限和角色绑定，避免产生不可见但仍被授权的孤立权限。

## 7. 职责分离（SoD）

路径：侧边栏 → 系统管理 → 职责分离。

### SSD 静态互斥

SSD 用于禁止同一用户同时拥有一组互斥角色。例如“审计员”和“开发人员”不应由同一人同时承担。

系统在角色分配阶段检查 SSD。若保存失败：

1. 查看提示中的冲突角色。
2. 取消其中一个角色，或由安全负责人审阅约束。
3. 不要直接修改 sys_user_role 绕过检查。

### DSD 动态互斥

DSD 允许用户拥有相关角色，但禁止在同一会话同时激活。例如经理与 HR 可以分配给同一人，但要分开会话职责。

系统在登录默认激活和顶栏角色切换时检查 DSD。

### 维护约束

创建或编辑约束时：

- 名称应说明业务风险。
- 至少选择两个角色。
- 明确选择 SSD 或 DSD。
- 修改后用受约束用户做一次允许路径和拒绝路径测试。

仓库中的内置约束是演示基线，不等同于组织正式制度。

## 8. 部门管理

路径：侧边栏 → 系统管理 → 部门管理。

页面左侧是部门树，右侧显示部门详情和成员。

新增或编辑字段：

| 字段 | 必填 | 说明 |
|---|---|---|
| 上级部门 | 否 | 留空表示根节点 |
| 部门名称 | 是 | 同层级内应清晰可辨 |
| 联系电话 | 否 | 部门电话 |
| 邮箱 | 否 | 部门邮箱 |
| 排序 | 否 | 数字越小越靠前 |
| 状态 | 是 | 正常或停用 |

当前界面不提供“负责人”选择项；不要依据旧手册寻找该字段。

## 9. 日志

### 登录日志

路径：侧边栏 → 日志管理 → 登录日志。

记录用户名、IP、浏览器、操作系统、时间、成功/失败状态和消息。系统对不存在用户与密码错误使用统一外部提示，但审计记录仍可供授权管理员排查。

### 操作日志

路径：侧边栏 → 日志管理 → 操作日志。

记录受审计切面覆盖的管理操作，包括模块、请求方法、操作者、IP、时间、结果和耗时。它不是所有 HTTP 请求的完整访问日志。

查看请求参数和结果时注意其中可能包含个人信息。删除日志前应符合组织留存要求。

## 10. 个人资料

右上角个人菜单支持：

- 修改个人资料。
- 修改当前密码。
- 上传头像。
- 退出登录。

头像必须是图片，大小不超过 2 MB。Docker 环境若上传成功但图片无法显示，应由运维检查 /uploads/ 反向代理和持久化挂载。

## 11. 常见问题

### 登录后看不到菜单

1. 检查当前激活角色。
2. 检查角色是否启用。
3. 检查角色是否有对应 system:*:query 权限。
4. 重新登录以获取最新令牌。

### 分配权限后仍无法操作

1. 检查当前会话是否激活了对应角色。
2. 检查权限和角色状态。
3. 检查 SSD/DSD。
4. 若数据库被脚本直接更新，确认同一事务已递增受影响用户的 `auth_version`；使用规范的 `init-permissions.sql` 时会自动全局递增。无需清理 Redis，禁止执行 `FLUSHDB`/`FLUSHALL`。

### 忘记密码

联系具有 user:resetPassword 权限的管理员。系统不提供通过邮件或短信自助找回。

### Swagger 无法匿名访问

这是默认安全行为。只有开发者可在可信 dev 环境临时设置 app.security.public-docs=true；普通用户不应要求在生产环境开放。

### 需要技术支持

本仓库没有内置工单系统，也没有有效的公开客服电话或支持邮箱。请通过项目维护者约定的渠道提交问题，并附上时间、操作步骤、错误信息和已脱敏日志。
